# PRD-016 — bootstrap-cicd (relocate the gate to CI)

**User-visible-surface impact:** Yes — a new `bootstrap-cicd` skill. One command drops the repo's
EXISTING gate into the repo as `~~ci` (a CI workflow + a pre-push hook) plus an idempotent
managed block in `CLAUDE.md`. No behavior change to the loop itself.

## 1. Problem (with proof)

The framework already promises CI is the thing that makes its rules binding. `CONNECTORS.md`:

> "`~~ci` is what makes the constitution's invariants *enforced* rather than aspirational."

But the promise is undelivered. Verified in-repo:

```
$ ls .github/workflows        ->  (no .github/workflows)
$ grep -rn "stop_gate.py ci"  ->  (nothing — no standing/CI check entry point exists)
```

Every enforcement artifact built in Arc 1 — the Stop gate, the executor lock, the tamper-check,
the ledger — lives on **one operator's machine** and fires only when **that operator's agent ends
a turn** (`hooks/stop_gate.py` is a Claude Code `Stop` hook). Push directly and nothing runs; a
second collaborator never sees the gate. The moat is real but **bypassable and unshared**.

## 2. Root cause

Enforcement was built local-only by construction: the gate is a turn-end hook, not a
push-/merge-time check on a remote nobody controls. There is no generator that relocates the gate
to `~~ci`, and no single "standing checks" entry point a CI job could call — so even a hand-written
workflow would have to **re-list** the checks in YAML, forking the source of truth (the sin
`ARCHITECTURE.md` punishes hardest).

## 3. Scope

The smallest change that makes the existing gate runnable on a remote without forking the check
list, and a generator that installs it into any repo.

- **One standing-checks source + a vendor-able runner — extend the gate concept, don't fork the
  checks.** A new self-contained engine `hooks/ci_gate.py` runs a **standing** check set from
  `.orchestrator/ci-gate.json` (distinct from the per-PRD `gate.json`), with the same fail-closed
  semantics and the same ledger (`source: "ci-gate"`). `--fast` runs only checks flagged `fast`
  (the pre-push subset). It is **standalone** (no import of plugin internals) so `scaffold.py` can
  vendor it into a target repo and it runs on a clean CI runner where the plugin isn't installed.
  `stop_gate.py ci` delegates to it for ergonomics. The ONE place checks are enumerated is
  `.orchestrator/ci-gate.json` (data); the engine is the same file everywhere (copied, never
  re-listed).
- **`skills/bootstrap-cicd/scaffold.py`** — the generator. Emits, parameterized by `~~ci`/`~~vcs`:
  1. a CI workflow whose only job is to invoke `python3 hooks/ci_gate.py` (no inline check list);
     `--ci github` → `.github/workflows/orchestrator-gate.yml`, otherwise a portable POSIX
     `orchestrator-gate.sh` template;
  2. a `pre-push` hook (`~~vcs=git` → `.githooks/pre-push`; opt-in `core.hooksPath` left to the
     user, not silently set) running `python3 hooks/ci_gate.py --fast`;
  3. vendors `hooks/ci_gate.py` into the target repo if absent (so CI is self-sufficient);
  4. a **managed sentinel block** appended to the target `CLAUDE.md`
     (`<!-- ORCHESTRATOR-LOOP:CI:BEGIN -->` … `:END`), **idempotent** — re-running replaces the
     block in place, never duplicates, never touches content outside the sentinels.
- **`skills/bootstrap-cicd/SKILL.md`** (+ `references/methodology.md`) — runs the generator,
  explains "CI = the existing gate relocated," and carries the four on-thesis constraints.
- **The ratchet (minimal, but with baseline-trust).** `.orchestrator/ci-baseline.json` records a
  floor for one tracked signal; `stop_gate.py ci` fails if it regresses below the floor.
  `scaffold.py ratchet-baseline` **refuses to baseline a signal the ablation control does not
  move** — you cannot ratchet a number you have not proven is real (per the app-profile sanity
  bound: a rising number after we shrink the rules is "too good" until the control reproduces it).
- **Constitution + catalog.** `ARCHITECTURE.md` gains a "Scaffolding" section (scaffold.py,
  ci-gate.json, ci-baseline.json, the `ci` runner); `CONNECTORS.md` `~~ci` row moves from
  "referenced" to "generated by bootstrap-cicd."

## 4. Non-goals

- **Does not activate CI.** Files are committed locally; the workflow only runs once the owner
  pushes. A push is the release call (app-profile) — out of scope here.
- Not a multi-provider matrix — GitHub workflow + one portable template; GitLab/CircleCI later.
- Not a multi-metric ratchet suite — one tracked signal as the reference implementation.
- Not the multi-hand release gate (that is 022) and not branch protection.

## 5. Acceptance tests (un-gameable)

- **AT-1 (no forked checks).** The generated workflow invokes `python3 hooks/ci_gate.py` and
  contains **no** inline check commands: `grep -E "self-test|check-startup|check-sync|pytest"
  <generated workflow>` returns nothing; the only check invocation is the single entry point.
  (Structurally impossible to pass by re-listing checks.)
- **AT-2 (the entry point really gates).** With a healthy tree `ci_gate.py` exits 0; with a planted
  failing check in `ci-gate.json` it exits nonzero and the failure is named on stderr AND appended
  to the ledger with `source: "ci-gate"`.
- **AT-3 (idempotent, non-destructive install).** Running `scaffold.py` twice yields a `CLAUDE.md`
  with **exactly one** managed block, byte-identical across runs, and all pre-existing content
  outside the sentinels unchanged (diff is empty on the second run).
- **AT-4 (app-agnostic parameterization).** `--ci github` emits `.github/workflows/…`; a non-github
  value emits the portable template; `grep -ri "github\|vercel\|supabase" skills/bootstrap-cicd/SKILL.md`
  finds no hardcoded product (skill refers to `~~ci`/`~~vcs` only).
- **AT-5 (baseline-trust).** `scaffold.py ratchet-baseline` **refuses** (nonzero, no file written)
  when handed a signal whose ablation control does not move it; **accepts** when the control shows a
  delta. A ratchet baseline cannot be set on an unproven number.
- **AT-6 (fast gate blocks a bad push).** The installed `pre-push` hook runs `ci_gate.py --fast`; a
  red fast check makes the hook exit nonzero (push refused) — proven by invoking the hook script
  directly.

All six are reproducible locally with no push and no GitHub round-trip.

## 6. Architect review

1. **Removal.** Removes the gap between `CONNECTORS.md`'s promise and reality; replaces the implicit
   "`~~ci` is a future connector" placeholder with a real generator. Nothing destructive removed.
2. **Single source of truth.** The standing checks live once, in `.orchestrator/ci-gate.json`, run
   by `hooks/ci_gate.py`; the CI workflow and the pre-push hook both call that one engine — no
   second list in YAML. **Ledger-writer note (disclosed):** the ledger keeps a single *writer
   concept* — the gate — but this PRD gives it a second entry point (`ci_gate.py` beside
   `stop_gate.py`), append-only and atomic for the small entries it writes; full cross-writer
   concurrency is hardened in 018, not forked into a second ledger. Version stays the one manifest
   pair.
3. **Layering.** EXTENDS `hooks/stop_gate.py` with a `ci` runner (reusing `run_checks`/`append_ledger`)
   rather than adding a parallel CI runner; EXTENDS `CLAUDE.md` via a managed block, not a second
   config file.
4. **Migration debt.** New components are catalogued in `ARCHITECTURE.md` in this PRD; the
   `CONNECTORS.md` `~~ci` row is updated in the same change. Ships its own cleanup.
5. **Constitution diff.** EXTENDS — adds a "Scaffolding" section to `ARCHITECTURE.md`; does not fork
   the gate. Cites the existing "Stop gate" and "Hooks" constitution entries it builds on.

Passes — proceed to handoff (self/solo build).

## 7. Execution

Tests-first: `test/harness/test_scaffold.py` + the `ci`-runner / ratchet tests written and committed
**failing**, baseline recorded, then implemented to green. Commit locally as `PRD-016` once
AT-1…AT-6 pass. **No push** — activation (and the version bump that ships it) is the owner's release
call. No migration/DDL in this PRD.
